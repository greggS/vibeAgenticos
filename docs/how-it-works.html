AgenticOS

An AI-first Linux desktop where every response from the agent is a fully designed website replacing the entire screen.

🧠What Is AgenticOS?

AgenticOS replaces the traditional desktop (icons, taskbar, file manager) with two elements: a fullscreen Agents Zone that the AI designs on each response, and a minimal Prompt Bar where the user types.

The AI agent doesn't output chat bubbles — it outputs raw HTML pages. Each response becomes a custom interface: dashboard, animation, diagram, game, or anything else it imagines. The screen updates live as new responses arrive.

Wayland Desktop

labwc compositor on Ubuntu 24.04 ARM64 (VirtualBox VM on M2 Mac). Screen is 1280×800px.

OpenClaw Gateway

Self-hosted AI agent runtime. Connects to Grok AI. Exposes a WebSocket API on port 18789.

Node.js Bridge

Authenticates with OpenClaw, extracts HTML from final responses, writes the UI file, triggers reload.

Chromium × 2

Two separate Chromium windows: agents zone (720px kiosk) and prompt bar (80px, always on top).

The return path is where the magic happens:

  1. OpenClaw streams event:"agent" events back through the bridge as the model generates tokens.

  2. When generation finishes, OpenClaw emits a event:"chat", payload:{state:"final"} event containing the complete message.

  3. The bridge intercepts this event, extracts any HTML/SVG from the text, and writes it to ~/.openclaw/ui-custom/agents-only.html.

  4. After 150ms (to ensure the file is flushed), the bridge broadcasts {type:"reload"} to all connected browsers.

  5. The Agents Zone browser receives the reload signal and calls location.reload(), loading the freshly written HTML.

🔌The Bridge (openclaw-bridge.js)

The bridge is a Node.js process that sits between Chromium and OpenClaw. It handles two concerns: authentication (complex, OpenClaw-specific) and HTML extraction + reload (the core AgenticOS feature).

Authentication Flow

OpenClaw uses Ed25519 device signatures. The bridge:

  1. Connects to ws://127.0.0.1:18789 with Origin: http://127.0.0.1:18789.

  2. Receives a connect.challenge event with a one-time nonce.

  3. Builds a signing message: "v2|deviceId|openclaw-control-ui|webchat|operator|scopes|timestamp|deviceToken|nonce"

  4. Signs it with the device's Ed25519 private key and sends a connect request.

  5. On success, flushes any queued messages from browsers.

HTML Extraction

When a chat.final event arrives, handleFinalMessage() extracts the text content and tries three patterns in order:

function extractHtml(text) {
  // 1. Fenced code block: ```html ... ```
  let m = /```html\s*\n([\s\S]*?)```/i.exec(text);
  if (m) return m[1].trim();

  // 2. Raw HTML document: <!DOCTYPE html> ... </html>
  m = /(<(?:!DOCTYPE\s+html|html)\b[\s\S]*?<\/html>)/i.exec(text);
  if (m) return m[1].trim();

  // 3. SVG block — wrapped in a dark full-screen page
  m = /(<svg\b[\s\S]*?<\/svg>)/i.exec(text);
  if (m) return `<!DOCTYPE html>...${m[1]}...</html>`;

  return null; // falls back to styled text page
}

If no HTML is found, the full text response is wrapped in a styled dark page with readable typography.

Script Injection

Every page written by the bridge — whether agent HTML or the fallback text page — gets a small WebSocket script injected before </body>. This script:

Reconnects to ws://127.0.0.1:18790 on close Calls location.reload() on {type:"reload"} Shows "thinking…" on lifecycle start events

Charset Fix

Agent-generated HTML often omits <meta charset="UTF-8">, causing emojis and special characters to render as garbled Latin-1. The bridge injects the charset tag into <head> if it's missing:

function ensureCharset(html) {
  if (/<meta[^>]+charset/i.test(html)) return html;
  const tag = '<meta charset="UTF-8">';
  if (html.includes("<head>")) return html.replace("<head>", "<head>\n" + tag);
  // fallbacks for <head ...> and headless HTML
  ...
}

Reconnect Behaviour

If OpenClaw disconnects (restart, crash, rate limit), the bridge reconnects with exponential backoff starting at 5s, capping at 60s. Browser clients get a {type:"bridge.status", status:"disconnected"} message while waiting.

🌐UI Files

Both UI files are served by python3 -m http.server 8080 from ~/.openclaw/ui-custom/. This avoids file:// restrictions with WebSocket connections.

agents-only.html — The Default State

This file is the initial page loaded when the system starts. It shows a minimal waiting indicator and connects to the bridge WebSocket. Once the agent responds, this file is overwritten by the bridge with the agent's HTML, and the browser reloads into it. From that point forward, the file changes on every agent response.

💡

Key insight: The agents zone never stays on the same HTML file twice. Each response is a completely fresh HTML document — the agent designs the entire interface from scratch each time.

prompt-only.html — The Prompt Bar

A persistent dark-themed input bar. Sends messages to the bridge using this format:

{
  "type": "req",
  "id": "unique-id",
  "method": "chat.send",
  "params": {
    "sessionKey": "main",
    "message": "user text here",
    "deliver": false,
    "idempotencyKey": "unique-id"
  }
}

The bridge forwards this directly to OpenClaw's WebSocket. The prompt bar also receives bridge.status events and shows a connection indicator dot.

📜Agent Instructions (AGENTS.md)

OpenClaw reads ~/.openclaw/workspace/AGENTS.md at the start of every session and injects it as system context. This is how the agent knows to output HTML instead of text.

Key instructions in the file:

Output raw <!DOCTYPE html> — no code fences Fill 1280×720px — use the whole screen Dark theme: bg #0a0e1a, accent #6366f1 Design-first: dashboards, not chat bubbles 
## OUTPUT FORMAT — NON-NEGOTIABLE

Every response must be raw HTML — output it directly, no code fences.
Start with <!DOCTYPE html> and end with </html>.

TEMPLATE:
<!DOCTYPE html>
<html>
<head><style>
* { margin:0; padding:0; box-sizing:border-box; }
body { width:1280px; height:720px; overflow:hidden;
       background:#0a0e1a; color:#e8eaf0; }
</style></head>
<body>
  <!-- YOUR FULL SCREEN DESIGN HERE -->
</body></html>
⚠️

Why no code fences? If the agent wraps HTML in ```html fences, the bridge still extracts it correctly — but asking the agent to output raw HTML means the model's entire output token budget goes to the interface, not wrapper text.

🚀Startup Sequence

labwc runs ~/.config/labwc/autostart on Wayland session start. The sequence is carefully ordered with sleeps to let each service come up before dependents start:

  1. 3s delay — wait for the Wayland session to fully initialize.

  2. HTTP server — python3 -m http.server 8080 from ~/.openclaw/ui-custom/. Serves both HTML files.

  3. Bridge — node openclaw-bridge.js. Starts listening on port 18790 immediately, then connects to OpenClaw asynchronously. Logs to /tmp/bridge.log.

  4. 4s delay — give the bridge time to authenticate before Chromium loads and connects.

  5. Agents Chromium — kiosk mode, 1280×720, loads agents-only.html, separate user profile at /tmp/chromium-agents.

  6. 2s delay — stagger the second Chromium launch.

  7. Prompt Chromium — 1280×80 at y=720, loads prompt-only.html, separate user profile at /tmp/chromium-prompt.

💡

Separate user profiles are critical. Without --user-data-dir set differently per window, Chromium detects a running singleton and opens the second URL as a tab inside the first kiosk window instead of a new window.

OpenClaw gateway itself (openclaw gateway) runs as a system service and is expected to already be up when labwc starts.
